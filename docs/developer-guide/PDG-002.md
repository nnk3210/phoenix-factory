# 1. Purpose

This document defines the engineering principles governing every software component developed under Project Phoenix.

These principles are mandatory for every module, service, engine, pipeline, plugin, and tool contained within the Phoenix ecosystem.

No implementation may violate these principles without an approved architecture decision.

---

# 2. Engineering Philosophy

Project Phoenix follows Specification-Driven Engineering.

Every implementation shall begin with an approved specification.

Architecture defines implementation.

Implementation never defines architecture.

Engineering decisions must prioritize long-term maintainability over short-term convenience.

---

# 3. Core Engineering Principles

Every software component shall satisfy the following principles.

## 3.1 Maintainability

Code shall remain understandable and maintainable over long periods.

Readability has higher priority than clever implementations.

---

## 3.2 Extensibility

Every module shall support future extension without requiring modification of existing business logic.

---

## 3.3 Testability

Business logic shall be independently testable.

Hidden dependencies are prohibited.

---

## 3.4 Reusability

Reusable components shall be extracted into dedicated modules.

Duplicated logic is prohibited.

---

## 3.5 Deterministic Behavior

The same input shall always produce the same output unless explicitly documented.

---

## 3.6 Simplicity

Solutions shall remain as simple as possible while satisfying all functional requirements.

---

## 3.7 Consistency

Naming, architecture, interfaces, logging, and documentation shall remain consistent throughout the entire project.

---

# 4. Architecture Principles

Project Phoenix adopts layered enterprise architecture.

Each layer owns exactly one responsibility.

Layers communicate only through well-defined interfaces.

Direct dependencies between unrelated layers are prohibited.

The architecture must support:

- Replacement
- Isolation
- Independent Testing
- Independent Deployment

---

# 5. Code Quality Principles

Every code contribution shall satisfy the following requirements.

- Self-explanatory naming
- Small functions
- Single Responsibility
- No duplicated business logic
- Explicit error handling
- Complete logging
- Minimal coupling
- High cohesion

Complexity shall always be reduced whenever possible.

---

# 6. Development Workflow

Every implementation shall follow the workflow below.

Requirement

↓

Specification

↓

Architecture

↓

Implementation

↓

Testing

↓

Review

↓

Release

No implementation shall bypass this workflow.

---

# 7. Documentation Principles

Documentation is considered part of the software product.

Every significant engineering decision shall be documented.

Documentation shall evolve together with implementation.

Code without documentation is considered incomplete.

---

# 8. Testing Principles

Every critical component shall support automated testing.

Testing categories include:

- Unit Testing
- Integration Testing
- Pipeline Testing
- Regression Testing

Manual testing alone is insufficient.

---

# 9. Version Control Principles

Every significant milestone shall be committed independently.

Commit messages shall clearly describe completed work.

History must remain understandable.

Direct modification of released versions is prohibited.

---

# 10. Decision Making

When multiple technical solutions exist, priority shall follow the order below.

1. Correctness
2. Maintainability
3. Readability
4. Extensibility
5. Performance

Performance optimization shall never sacrifice architecture quality unless objectively justified.
